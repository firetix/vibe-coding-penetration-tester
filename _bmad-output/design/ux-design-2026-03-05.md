# VibePenTester UX Design Specification (MVP)

## 1) UX Intent
Create a premium, trustworthy, dark-first security product experience that helps users move from uncertainty to confidence:
1. **Understand risk quickly**
2. **Act on findings confidently**
3. **Upgrade when value is proven**

Design principles:
- **Signal > noise**: prioritize actionable findings over technical clutter
- **Fast orientation**: users always know where they are and what to do next
- **Progressive detail**: summary first, deep detail on demand
- **Trust through clarity**: explicit statuses, evidence, and audit-friendly output

## 2) Primary User Flows
1. **Acquisition flow:** Landing → CTA → Signup/Login
2. **Activation flow:** Auth → Create first scan → Watch live updates → Review findings
3. **Conversion flow:** Hit entitlement/paywall state → Understand value gap → Upgrade
4. **Retention flow:** Return to scans list → Compare outcomes → Export report

## 3) Information Architecture
### Public
- `/` Landing
- `/login`
- `/signup`

### Authenticated app shell
- `/app` Dashboard summary (lightweight)
- `/app/scans` Scans list + create form
- `/app/scans/[id]` Scan detail with live event stream
- `/app/reports` Report list/download
- `/app/billing` Plan + usage + upgrade

## 4) Visual Language & Tokens
- **Theme:** dark-first
- **Background:** `#0A0A0F`
- **Primary accent:** cyan `#06B6D4`
- **Success:** `#10B981`
- **Warning:** `#F59E0B`
- **Critical/error:** `#EF4444`
- **Typography:** Inter (UI), JetBrains Mono (technical IDs/logs)
- **Surface style:** subtle glass cards + soft border contrast

Motion guidelines:
- quick, purposeful transitions (120–220ms)
- avoid decorative motion during active scan updates

## 5) Page Specifications
## 5.1 Landing (`/`)
### User goal
Determine whether VibePenTester is credible and worth trying now.

### Layout blocks
1. Hero: headline, one-line value, primary CTA “Scan your app”, secondary CTA “See demo”
2. Trust strip: positioning + social proof (logos/stars/testimonials)
3. How it works (3-step)
4. Feature grid (scan depth, live view, reports, billing clarity)
5. Pricing cards (Free, Pro $19)
6. FAQ
7. Final CTA

### Critical interactions
- URL demo input with immediate validation state
- “Start free” always visible above fold
- pricing CTA routes to auth or billing flow based on session

### States
- Demo unavailable fallback: show product screenshot + alternate CTA
- Slow network: skeletons on pricing/social proof blocks

## 5.2 Login (`/login`) and Signup (`/signup`)
### User goal
Authenticate with minimal friction and clear redirect behavior.

### Layout
- centered auth card with optional context panel
- email/password + social option if enabled
- clear secondary action link between login/signup

### UX requirements
- preserve intended destination via `redirectedFrom`
- inline validation errors; avoid generic “something went wrong”
- success redirect to `/app` or requested protected route

### Error states
- invalid credentials
- expired reset/session links
- provider failure (temporary)

## 5.3 Scans List (`/app/scans`)
### User goal
Create scans quickly and monitor recent scan activity.

### Layout
- header with page purpose + quick docs hint
- create scan form (target URL, mode)
- recent scans table/list

### Key components
- `TargetUrlInput`
- `ScanModeSelect`
- `CreateScanButton`
- `ScansTable` with status chip and “Live updates” link

### Interaction behavior
- optimistic button/loading state while creating
- success toast/banner on scan creation
- list refresh on successful create

### States
- Empty: educational prompt “Create your first scan”
- Error: inline alert with retry action
- Missing token/auth drift: explicit relogin guidance

## 5.4 Scan Detail + Live View (`/app/scans/[id]`)
### User goal
Observe scan progress in realtime and understand evidence quickly.

### Layout
1. breadcrumb/back link
2. scan metadata card (id/status/target/stream state)
3. live event timeline (chronological)
4. findings summary panel (as findings mature)
5. test/manual event tool (dev/testing context)

### Key components
- `StreamStateBadge`
- `EventTimeline`
- `EventPayloadViewer` (collapsible JSON)
- `FindingsSeveritySummary`
- `RetryConnectionNotice`

### Realtime behavior
- connect to SSE stream at load
- show explicit states: connecting, connected, reconnecting, disconnected
- use heartbeat-aware UI (not every heartbeat rendered to timeline)

### Failure handling
- stream drop: passive warning + auto-reconnect
- scan not found/not owned: blocked state with return CTA

## 6) Shared Component Map
- **Navigation/App shell:** SidebarNav, TopBar, UserMenu, ContextSwitcher
- **Data display:** Card, Table, StatusBadge, SeverityBadge, EmptyState
- **Inputs:** URLInput, Select, Button, InlineMessage
- **Realtime:** StreamStateBadge, TimelineItem, LiveIndicator
- **Monetization:** PlanCard, UsageMeter, UpgradeCTA, PaywallNotice
- **Reports:** ReportRow, DownloadAction, GenerationStatusChip

## 7) Copy and Content Tone
- concise, security-literate, non-alarmist
- avoid fear language; focus on actionable outcomes
- examples:
  - good: “2 high-severity findings need attention before release.”
  - avoid: “Your app is dangerously vulnerable!”

## 8) Accessibility & Responsiveness
- WCAG 2.1 AA baseline
- keyboard operable for all interactive controls
- visible focus rings and contrast-compliant status colors
- mobile-first responsive behavior:
  - scans table collapses to stacked cards on narrow widths
  - sticky primary actions on mobile where needed

## 9) UX Telemetry (for validation)
Track key events:
- `landing_cta_clicked`
- `auth_success`
- `scan_created`
- `scan_stream_connected`
- `scan_stream_reconnected`
- `finding_opened`
- `report_downloaded`
- `upgrade_cta_clicked`
- `upgrade_completed`

## 10) Acceptance Criteria (UX)
1. New user can reach first created scan in <= 3 minutes.
2. Scan detail stream state is always visible and understandable.
3. Errors expose immediate next action (retry, relogin, return).
4. Users can navigate primary flows on mobile and desktop without dead ends.
5. Pricing/upgrade intent is visible but does not interrupt active scan workflows.

## 11) Open UX Questions
1. Should scan detail timeline default to collapsed technical payloads for non-technical users?
2. Should `/app` dashboard stay minimal in MVP or include severity trend chart immediately?
3. Should billing page include proactive quota warnings before hard entitlement failure?
